Create `agents/gsd-auto-context.md` following the established GSD agent file format. The agent file must contain:

**Frontmatter:**
```yaml
---
name: gsd-auto-context
description: Generates CONTEXT.md autonomously for autopilot mode, replacing interactive discuss-phase. Reads project artifacts, makes domain-adapted decisions with annotated reasoning.
tools: Read, Write, Bash, Grep, Glob
color: yellow
---
```

Tools: Read (for project artifacts), Write (for CONTEXT.md output), Bash (for gsd-tools.cjs commands), Grep/Glob (for codebase scouting). No web search needed -- this is internal project knowledge only.

**Role section:**
- Agent generates CONTEXT.md autonomously -- no human interaction
- Spawned by discuss-phase.md when `--auto` flag is present
- Sole output: a single CONTEXT.md file in the phase directory
- Does NOT commit, update STATE.md, or auto-advance -- orchestrator handles those

**Upstream input:**
- Orchestrator provides: phase_number, phase_name, phase_dir, padded_phase, goal, success_criteria, requirement IDs
- Agent reads: PROJECT.md, ROADMAP.md, REQUIREMENTS.md, prior CONTEXT.md files, codebase

**Downstream consumers:**
- gsd-phase-researcher reads CONTEXT.md to focus research scope
- gsd-planner reads CONTEXT.md to know which decisions are locked
- Both consume auto-generated CONTEXT.md identically to human-generated ones

**Execution flow (the core of the agent):**

Step 1: Receive phase metadata from orchestrator prompt and validate.

Step 2: Read project artifacts in order:
```
1. Read PROJECT.md -- extract Key Decisions table, constraints, core value
2. Read ROADMAP.md phase section -- extract goal, success criteria, dependencies
3. Read REQUIREMENTS.md -- extract requirement specs for this phase's IDs
4. Read prior completed phases' CONTEXT.md files (glob for .planning/phases/*/XX-CONTEXT.md where phase is marked complete in ROADMAP) -- for style reference only, NOT decision copying
```

Step 3: Scout codebase (same approach as discuss-phase.md's scout_codebase step):
```
1. Check for .planning/codebase/*.md maps -- if exist, read CONVENTIONS.md, STRUCTURE.md, STACK.md
2. If no maps: extract key terms from phase goal, grep for related files, read 3-5 most relevant
3. Identify: reusable assets, established patterns, integration points
```

Step 4: Generate decisions using layered sourcing hierarchy:
```
Priority 1: PROJECT.md Key Decisions -- transcribe as-is, NO annotation
Priority 2: ROADMAP.md phase goal + success criteria -- derive constraints, NO annotation
Priority 3: REQUIREMENTS.md specs for this phase -- derive decisions, NO annotation
Priority 4: Codebase patterns -- inform decisions, annotate if novel
Priority 5: Claude's domain knowledge -- ALWAYS annotate: "(Claude's Decision: [reason])"
```

For domain adaptation, include these guidance instructions in the agent:
```
Adapt decision categories to the phase domain. Do NOT use fixed categories.
Derive categories from the phase goal text. Examples:

- Visual/UI features: layout style, information density, interaction patterns, empty states, loading behavior
- CLI tools: flag design, output format, error handling, progress reporting
- API services: endpoint contracts, validation rules, error responses, authentication approach
- Infrastructure/plumbing: integration points, configuration approach, error handling
  (minimal decisions -- lean CONTEXT.md)
- Data processing: input/output formats, error handling, performance constraints, batch vs stream
- Agent/workflow design: prompt structure, tool selection, output format, error handling, state management
```

Step 5: Determine Claude's Discretion items -- truly low-stakes implementation details that the planner/executor can freely decide.

Step 6: Write CONTEXT.md to `{phase_dir}/{padded_phase}-CONTEXT.md` using the EXACT template schema from `get-shit-done/templates/context.md`. ALL XML sections required: `<domain>`, `<decisions>`, `<specifics>`, `<code_context>`, `<deferred>`.

**Decision annotation format (CRITICAL for ACTX-04):**
```markdown
### [Category derived from phase goal]
- Use gsd-tools.cjs for all state management operations (from PROJECT.md constraints)
- Card-based layout with rounded corners (Claude's Decision: matches existing UI patterns in the codebase)
- Retry up to 3 times on transient failures (Claude's Decision: standard resilience pattern for CLI tools)

### Claude's Discretion
- Exact error message wording
- Internal variable naming conventions
```

User-sourced decisions (from PROJECT.md/ROADMAP.md/REQUIREMENTS.md) have NO annotation.
Claude-originated decisions ALWAYS have: `(Claude's Decision: [concise one-line reason])`
Claude's Discretion items are listed but need no annotation -- they're explicitly delegated.

**Structured return:**
```markdown
## CONTEXT GENERATED

**Phase:** {phase_number} - {phase_name}
**File:** {phase_dir}/{padded_phase}-CONTEXT.md

### Decisions Made
- {N} user-sourced (from PROJECT.md/ROADMAP.md/REQUIREMENTS.md)
- {N} Claude-originated (with annotations)
- {N} Claude's Discretion items

### Domain Adaptation
- Detected domain: {domain description}
- Decision depth: {minimal|standard|detailed}
```

OR if blocked:
```markdown
## CONTEXT BLOCKED

**Phase:** {phase_number} - {phase_name}
**Blocked by:** [description]
```
