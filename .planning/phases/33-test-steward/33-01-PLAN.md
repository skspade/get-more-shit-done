**Frontmatter:**
```yaml
---
name: gsd-test-steward
description: Analyzes test suite health -- redundancy detection, staleness detection, budget enforcement, and consolidation proposals. Read-only analysis agent spawned by audit-milestone and audit-tests.
tools: Read, Bash, Grep, Glob

---
```

**Agent structure:**

1. `<role>` section:
   - Identity: Read-only test suite health analyzer
   - Spawned by: audit-milestone workflow (step 3.5) and /gsd:audit-tests command
   - Core responsibilities: budget analysis, redundancy detection, staleness detection, consolidation proposals
   - Critical constraint: NEVER modifies test files, NEVER auto-applies changes, only produces markdown report
   - Mandatory initial read instruction (same pattern as other agents)

2. `<process>` section with these steps:

   **Step 1: Load Input Data**
   - Read files listed in `<files_to_read>` block from orchestrator prompt
   - Use `findTestFiles()` from testing.cjs (via `gsd-tools.cjs test-count`) to discover all test files
   - Use `getTestConfig()` (via `gsd-tools.cjs test-config`) to read budget thresholds
   - Collect project test count via `gsd-tools.cjs test-count --raw`

   **Step 2: Budget Analysis**
   - Compare project test count against project budget (default 800)
   - If `--phase N` provided, compare phase test count against per-phase budget (default 50)
   - Calculate percentage used: `count / budget * 100`
   - Determine status: OK (< 80%), Warning (>= 80%), Over Budget (>= 100%)
   - Format budget section for report

   **Step 3: Redundancy Detection**
   - Read each test file content
   - For each file, extract assertion patterns: `test('...', ...)` and `it('...', ...)` descriptions plus the assertion bodies
   - Compare across files to find:
     a. **Duplicate assertions:** Same function under test, same input values, different test names
     b. **Overlapping coverage:** Integration test that exercises the exact same code path as existing unit tests
     c. **Near-duplicate tests:** Tests with >80% similar assertion patterns (same describe block, similar test names)
   - Use regex-based comparison consistent with testing.cjs counting approach
   - For each finding, record: file paths, test names, similarity description

   **Step 4: Staleness Detection**
   - For each test file, extract referenced source file paths and function names from:
     a. `require()` and `import` statements (the module being tested)
     b. Function names in assertion calls
   - Check if referenced source files still exist using `fs.existsSync` (via Grep/Glob on the codebase)
   - Check if referenced function names still exist in the source files
   - Flag tests where the primary module under test has been deleted or the tested function no longer exists

   **Step 5: Generate Report**
   - Produce a markdown report with these sections:

   ```markdown
   ## Test Suite Health Report

   **Generated:** {date}
   **Project:** {cwd}

   ### Budget Status

   | Metric | Count | Budget | Usage | Status |
   |--------|-------|--------|-------|--------|
   | Project | {N} | {budget} | {pct}% | {OK/Warning/Over} |
   | Phase {X} | {N} | {budget} | {pct}% | {OK/Warning/Over} |

   ### Redundancy Findings

   {If none: "No redundant tests detected."}

   {For each finding:}
   **{Type}: {description}**
   - Files: `{file1}`, `{file2}`
   - Tests: `{test1}`, `{test2}`
   - Similarity: {description}

   ### Stale Test Findings

   {If none: "No stale tests detected."}

   {For each finding:}
   **Stale: {test file}**
   - References: `{missing module or function}`
   - Reason: {file deleted / function removed}

   ### Consolidation Proposals

   {If none: "No consolidation proposals."}

   {For each proposal:}
   #### Proposal {N}: {strategy} — {title}
   - **Strategy:** {parameterize | promote | prune | merge}
   - **Source:** `{file(s)}`
   - **Action:** {what to do}
   - **Rationale:** {why}
   - **Estimated reduction:** {N} test(s)

   ### Summary

   - Budget: {status}
   - Redundant tests: {N} finding(s)
   - Stale tests: {N} finding(s)
   - Consolidation proposals: {N}
   ```

3. `<output>` section:
