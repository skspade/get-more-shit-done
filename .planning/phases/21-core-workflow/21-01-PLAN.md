    Create `get-shit-done/workflows/linear.md` following the established workflow structure (`<purpose>`, `<required_reading>`, `<process>`, `<success_criteria>`).

    The workflow must implement these steps:

    **Step 1: Parse arguments (WKFL-01)**
    - Extract issue IDs from `$ARGUMENTS` using pattern matching for tokens like `LIN-123`, `ENG-456` (any letter-dash-number pattern: `/[A-Z]+-\d+/gi`)
    - Extract flags: `--quick`, `--milestone`, `--full` from `$ARGUMENTS`
    - If both `--quick` and `--milestone` are present, error: "Cannot use both --quick and --milestone flags"
    - If no issue ID found, prompt user via AskUserQuestion with header "Linear Issue", question "Enter the Linear issue ID (e.g., LIN-123):", followUp null
    - Store as `$ISSUE_IDS` (array), `$FORCE_QUICK` (boolean), `$FORCE_MILESTONE` (boolean), `$FULL_MODE` (boolean)

    **Step 2: Fetch issue data (WKFL-02)**
    - For each issue ID in `$ISSUE_IDS`:
      - Call `mcp__plugin_linear_linear__get_issue` with `id: issue_id, includeRelations: true`
      - Call `mcp__plugin_linear_linear__list_comments` with `issueId: issue.id` (using the UUID from get_issue)
      - Store results in `$ISSUES` array with structure: `{id, identifier, title, state, labels, description, comments, children, relations}`
    - Display banner:
      ```
      ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
       GSD ► LINEAR ISSUE FETCHED
      ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
      ```
    - For each issue display: title, state, labels (comma-separated), comment count
    - On fetch failure, error with: "Failed to fetch issue {ISSUE_ID}: {error message}"

    **Step 3: Route via heuristic (WKFL-03, WKFL-04)**
    - If `$FORCE_QUICK`: set `$ROUTE = "quick"`, display "Route: QUICK (flag override)", skip scoring
    - If `$FORCE_MILESTONE`: set `$ROUTE = "milestone"`, display "Route: MILESTONE (flag override)", skip scoring
    - Otherwise compute `$MILESTONE_SCORE` starting at 0:
      - Multiple issues (length > 1): +3 points
      - Any issue has sub-issues/children: +2 points
      - Any issue description > 500 words: +1 point
      - Any label matching "feature" or "epic" (case-insensitive): +2 points
      - Any label matching "bug", "fix", "chore", "docs" (case-insensitive): -1 point
      - Any blocking or related relations: +1 point
    - If `$MILESTONE_SCORE >= 3`: set `$ROUTE = "milestone"`
    - If `$MILESTONE_SCORE < 3`: set `$ROUTE = "quick"`
    - Display: "Routing: {ROUTE} (score: {MILESTONE_SCORE})"

    **Step 4: Write linear-context.md**
    - Write `.planning/linear-context.md` with:
      ```markdown
      ---
      issue_ids: [{comma-separated issue identifiers}]
      route: {quick|milestone}
      score: {MILESTONE_SCORE or "override"}
      fetched: {ISO date}
      ---
      # Linear Context

      Issues fetched for this workflow run.
      Consumed by Phase 22 completion loop.
      ```

    **Step 5: Execute route**

    **If `$ROUTE == "quick"` (WKFL-05):**
    - Synthesize `$DESCRIPTION` from issue data:
      - Concatenate: first issue title + "\n\n" + first issue description (truncated to 1500 chars) + "\n\nLinear comments:\n" + first 3 comments body text (each truncated to 200 chars)
      - Total truncated to 2000 chars
      - For multiple issues, use first issue only for description (multiple issues route to milestone anyway due to +3 score)
    - Generate slug: `node "$HOME/.claude/get-shit-done/bin/gsd-tools.cjs" generate-slug "$DESCRIPTION"`
    - Initialize: `INIT=$(node "$HOME/.claude/get-shit-done/bin/gsd-tools.cjs" init linear)`
    - Parse init JSON for quick task fields
    - Then execute quick workflow steps 2-8 inline:
      - Step 2: Use init data (already have it)
      - Step 3: `mkdir -p "${quick_dir}"`
      - Step 4: `mkdir -p ".planning/quick/${next_num}-${slug}"`, set `$QUICK_DIR`
      - Step 5: Spawn planner in quick mode with description and Linear issue ID context appended: "Linear issue: {ISSUE_ID} -- {title}"
      - Step 5.5: If `$FULL_MODE`, run plan-checker loop (max 2 iterations)
      - Step 6: Spawn executor
      - Step 6.5: If `$FULL_MODE`, spawn verifier
      - Step 7: Update STATE.md quick tasks table. If table lacks "Linear" column, add it to header and separator. Add Linear issue ID to the new row.
      - Step 8: Commit and display completion
    - Use models from init JSON (planner_model, executor_model, checker_model, verifier_model)
    - Spawn planner as: `Task(subagent_type="gsd-planner", model="{planner_model}")`
    - Spawn executor as: `Task(subagent_type="gsd-executor", model="{executor_model}")`
    - If full mode, spawn checker as: `Task(subagent_type="gsd-plan-checker", model="{checker_model}")`
    - If full mode, spawn verifier as: `Task(subagent_type="gsd-verifier", model="{verifier_model}")`

    **If `$ROUTE == "milestone"` (WKFL-06):**
    - Build MILESTONE-CONTEXT.md from issue data:
      ```markdown
      # Milestone Context

      **Source:** Linear issues ({comma-separated identifiers})

      ## Features
      {For each issue: "### {title}\n{description}\n"}

      ## Additional Context
      {For each issue with comments: key comments summarized}
      ```
    - Write to `.planning/MILESTONE-CONTEXT.md`
    - Initialize milestone models: `MINIT=$(node "$HOME/.claude/get-shit-done/bin/gsd-tools.cjs" init new-milestone)`
    - Parse milestone init JSON for roadmapper_model, researcher_model, synthesizer_model
    - Execute new-milestone workflow steps 1-11 inline with MILESTONE-CONTEXT.md pre-populated
    - Use models from milestone init JSON for spawned agents

      ```
      Issue: {ISSUE_ID} — {title}
      Route: Quick task
      Directory: {QUICK_DIR}
      ```
    - Milestone route:
