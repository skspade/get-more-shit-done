# Plan 18-01: Settings Command

## Objective

Replace the `handleSettings` stub in `cli.cjs` with a real implementation that lets users view all config settings, set individual values with validation, and receive clear error messages for invalid values.

## must_haves

- `gsd settings` displays all config.json key-value pairs with nested keys flattened to dot notation
- `gsd settings set <key> <value>` writes a validated value to config.json
- `gsd settings set <key> <invalid-value>` shows a validation error and does NOT write to disk
- JSON and plain output modes work correctly via `formatOutput()`
- All existing tests continue to pass

## Verification

After implementation, run:
```bash
node tests/cli.test.cjs
```
All tests must pass with zero failures.

Manual verification:
```bash
node get-shit-done/bin/gsd-cli.cjs settings --json
node get-shit-done/bin/gsd-cli.cjs settings set model_profile balanced
node get-shit-done/bin/gsd-cli.cjs settings set model_profile invalid_value
```

<task id="1" title="Implement gatherSettingsData and handleSettings view mode">

**What:** Add `gatherSettingsData(projectRoot)` function and replace the `handleSettings` stub with a real implementation that handles the view mode (`gsd settings` with no subcommand).

**Where:** `get-shit-done/bin/lib/cli.cjs`

**How:**

1. Add a `gatherSettingsData(projectRoot)` function that:
   - Reads `.planning/config.json` from `projectRoot`
   - If the file does not exist, returns `{ error: true, message: 'config.json not found. Run /gsd:health --repair to create defaults.' }`
   - Flattens nested objects to dot-notation keys (e.g., `workflow.research`, `autopilot.circuit_breaker_threshold`)
   - Returns `{ settings: [...] }` where each entry is `{ key, value }`

2. Replace the `handleSettings()` function with `handleSettings(projectRoot, args)`:
   - If `args` is empty or `args[0]` is not `'set'`: run view mode
   - Call `gatherSettingsData(projectRoot)`
   - If error, return `{ command: 'settings', error: true, message: data.message }`
   - Build rich-mode output:
     - Header line: `Settings (.planning/config.json)` in bold
     - Blank line
     - For each setting: `  {DIM}{key}{RESET} = {BOLD}{value}{RESET}`
   - Return `{ command: 'settings', settings: data.settings, message }`

**ANSI constants** to define at top of function: `BOLD`, `RESET`, `DIM` (same pattern as `handleHealth`).

**Flatten logic:**
```javascript
function flattenConfig(obj, prefix) {
  const result = [];
  for (const [key, val] of Object.entries(obj)) {
    const fullKey = prefix ? `${prefix}.${key}` : key;
    if (val !== null && typeof val === 'object' && !Array.isArray(val)) {
      result.push(...flattenConfig(val, fullKey));
    } else {
      result.push({ key: fullKey, value: val });
    }
  }
  return result;
}
```

Define `flattenConfig` as a module-level helper (above `gatherSettingsData`).

</task>

<task id="2" title="Implement handleSettings set mode with validation">

**What:** Add the `set` subcommand path to `handleSettings` so that `gsd settings set <key> <value>` validates and writes to config.json.

**Where:** `get-shit-done/bin/lib/cli.cjs`

**How:**

1. In `handleSettings(projectRoot, args)`, when `args[0] === 'set'`:
   - Extract `key = args[1]` and `rawValue = args[2]`
   - If key or rawValue is missing, return `{ command: 'settings', error: true, message: 'Usage: gsd settings set <key> <value>' }`
   - Parse the value: `'true'`/`'false'` become booleans, numeric strings become numbers, else keep as string
   - Validate the key/value pair (see validation rules below)
   - If validation fails, return `{ command: 'settings', error: true, key, value: rawValue, message: validationErrorMessage }` -- do NOT write to disk
   - If validation passes, read config.json, set the value using dot-notation key traversal (same logic as `cmdConfigSet` in config.cjs), write back
   - If the key is unknown (not in the known keys list), print an info notice but still write the value
   - Return `{ command: 'settings', updated: true, key, value: parsedValue, message: confirmationMessage }`

2. Validation rules (implement as a `validateSetting(key, value)` function that returns `null` on success or an error string on failure):
   - `model_profile`: must be one of `'quality'`, `'balanced'`, `'budget'`
   - `commit_docs`, `search_gitignored`, `parallelization`: must be boolean
   - `branching_strategy`: must be one of `'none'`, `'phase'`, `'milestone'`
   - `workflow.research`, `workflow.plan_check`, `workflow.verifier`, `workflow.auto_advance`: must be boolean
   - `autopilot.circuit_breaker_threshold`: must be a positive integer (> 0)
   - All other keys: no validation (pass through)

3. Rich output formatting:
   - Success: `{GREEN}{key}{RESET} = {BOLD}{parsedValue}{RESET} (updated)` with GREEN defined as `'\x1b[32m'`
   - Unknown key info notice: `{DIM}Note: '{key}' is not a recognized setting{RESET}` on a line before the confirmation
   - Validation error: `{RED}Error:{RESET} {message}` with RED defined as `'\x1b[31m'`

Define `validateSetting` as a module-level helper (above `handleSettings`).

</task>
