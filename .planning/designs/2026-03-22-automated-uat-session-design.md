# Automated UAT Session — Design

**Date:** 2026-03-22
**Approach:** Claude-as-Tester via Chrome MCP

## UAT Test Source & Format

The automated UAT session reads from existing UAT.md files generated by the `verify-work` workflow. Each test has a name, expected behavior description, and result field. No changes to the UAT.md format are needed — the Claude agent interprets the `expected:` field as natural language instructions.

**For the agent to execute tests, two pieces of metadata are required at the milestone level:**

A `uat-config.yaml` file in `.planning/` provides the app's base URL and optional startup command:

```yaml
base_url: "http://localhost:3000"
startup_command: "npm run dev"  # optional — agent starts this if app isn't running
startup_wait_seconds: 10
browser: "chrome-mcp"           # primary
fallback_browser: "playwright"  # fallback for headless environments
```

**Test discovery:** The agent scans all phase directories for `*-UAT.md` files with `status: complete` (tests already defined from prior verify-work sessions). If no UAT.md files exist, the agent generates test scenarios from SUMMARY.md files using the same logic as verify-work's initialization step.

## Chrome MCP Execution Engine

The core execution engine is a Claude subagent spawned with Chrome MCP access. The agent receives:

1. The full UAT.md content (test list with expected behaviors)
2. The `uat-config.yaml` settings (base URL, startup command)
3. Instructions to execute each test sequentially

**Agent behavior per test:**

1. **Navigate** — Use `chrome_navigate` to the relevant page (inferred from test context or explicit URL in test description)
2. **Interact** — Use `chrome_click_element`, `chrome_fill_or_select`, `chrome_keyboard` to perform the test action described in the expected behavior
3. **Observe** — Use `chrome_screenshot` + `chrome_get_web_content` to capture the resulting state
4. **Judge** — Compare the observed state against the expected behavior description. The agent uses its multimodal understanding of the screenshot plus the DOM content to determine pass/fail
5. **Record** — Write the result (pass/fail with evidence) back to a structured output

**Screenshot evidence:** Each test produces a screenshot saved to `.planning/uat-evidence/{phase}-test-{N}.png`. Failed tests include both the screenshot and a text description of what was observed vs. expected.

**Session management:** The agent calls `tabs_context_mcp` at startup, creates a new tab via `tabs_create_mcp`, and operates in that tab throughout the session. Tab is closed on completion.

## Playwright Fallback Engine

When Chrome MCP is unavailable (headless CI, no browser session, Chrome MCP tools return errors), the system falls back to Playwright.

**Detection logic:** Before spawning the Chrome MCP agent, attempt `tabs_context_mcp`. If it fails or times out (5 seconds), switch to Playwright mode.

**Playwright execution:** A Claude subagent is spawned with Bash access. It:

1. Checks for Playwright installation (`npx playwright --version`), installs if missing (`npx playwright install chromium`)
2. For each UAT test, generates a minimal inline Playwright script that:
   - Navigates to the URL
   - Performs the described interaction (click, fill, etc.)
   - Takes a screenshot
   - Extracts page content for comparison
3. Runs the script via `node` and captures results
4. The agent still interprets pass/fail using its understanding of the expected behavior — Playwright is just the browser driver, not the assertion engine

**Key difference from Approach 2:** No persistent `.spec.ts` files are generated. The agent writes ephemeral scripts, runs them, and discards them. The agent remains the "tester" — Playwright is just its hands.

**Output format:** Identical to Chrome MCP mode — screenshots in `.planning/uat-evidence/`, structured results in the same format.

## Autopilot Integration

The automated UAT session is triggered at the **milestone level**, after all phases are complete and the milestone audit passes, but before milestone completion.

**New step in `autopilot.mjs` main flow:**

```
All phases complete
  → Milestone Audit (existing)
    → Audit passes
      → **Automated UAT Session** (NEW)
        → UAT passes → Milestone Completion (existing)
        → UAT fails → Gap Closure Loop → Re-audit → Re-UAT
```

**Implementation in `autopilot.mjs`:**

A new `runAutomatedUAT()` function is added after `runMilestoneAudit()` returns 0. It:

1. Checks for `.planning/uat-config.yaml` — if missing, skips UAT (not all projects have web UIs)
2. Optionally runs the startup command (`npm run dev`) if configured
3. Spawns a Claude session with the prompt `/gsd:uat-auto` (the new workflow)
4. Parses the exit code and result:
   - Exit 0 + all tests pass → proceed to milestone completion
   - Exit 0 + failures found → feed failures as gaps into `runGapClosureLoop()`
   - Exit non-zero → debug retry (same as other steps)

**Gap format for failures:** Failed UAT tests are written as gaps in a `MILESTONE-UAT.md` file that follows the same YAML gap format as UAT.md:

```yaml
- truth: "Comment appears immediately after submission"
  status: failed
  reason: "Automated UAT: screenshot shows empty comment list after submit"
  severity: major
  evidence: ".planning/uat-evidence/04-test-2.png"
```

This file is consumed by `plan-milestone-gaps` to create fix phases, just like audit gaps.

## UAT-Auto Workflow

A new workflow file at `workflows/uat-auto.md` defines the `/gsd:uat-auto` slash command. This is the prompt that the autopilot invokes via `claude -p`.

**Workflow steps:**

1. **Load config** — Read `.planning/uat-config.yaml` for base URL, startup command, browser preference
2. **Discover tests** — Scan phase directories for `*-UAT.md` files. If none exist, generate test scenarios from SUMMARY.md files across all milestone phases
3. **Detect browser** — Try Chrome MCP (`tabs_context_mcp`). If available, use Chrome MCP mode. If not, use Playwright fallback
4. **Start app** — If `startup_command` is configured and the app isn't already running (check by fetching `base_url`), start it and wait `startup_wait_seconds`
5. **Execute tests** — For each test in each UAT.md:
   - Navigate, interact, observe, judge (as described in Chrome MCP Engine section)
   - Record result with screenshot evidence
6. **Write results** — Create `.planning/MILESTONE-UAT.md` with:
   - Frontmatter: `status: passed | gaps_found`, timestamp, test counts
   - Full results table (test name, phase, status, evidence path)
   - Gaps section in YAML format (only for failures)
7. **Commit results** — Commit `MILESTONE-UAT.md` and evidence screenshots
8. **Exit** — Exit 0 with status in stdout for autopilot to parse

**The workflow does NOT:**
- Ask the user anything (fully autonomous)
- Modify source code (read-only testing)
- Run for more than a configurable timeout (default: 10 minutes)

## Evidence & Reporting

**Evidence directory structure:**

```
.planning/
  uat-evidence/
    {milestone}/
      {phase}-test-{N}.png         # Screenshot per test
      {phase}-test-{N}-fail.png    # Additional screenshot on failure (e.g., after retry)
  MILESTONE-UAT.md                 # Results summary
```

**MILESTONE-UAT.md format:**

```markdown
---
status: passed | gaps_found
milestone: v1.2
browser: chrome-mcp | playwright
started: 2026-03-22T10:30:00Z
completed: 2026-03-22T10:35:00Z
total: 12
passed: 11
failed: 1
---

## Gaps

- truth: "Comment appears immediately after submission in list"
  status: failed
  reason: "Automated UAT: comment form submits but list does not refresh"
  severity: major
  evidence: "uat-evidence/v1.2/04-test-2.png"
  observed: "Comment list shows 3 items before and after submit"
```

**Gitignore:** Evidence PNGs are committed to git (they're small, milestone-scoped, and valuable for debugging). Old evidence is cleaned up during milestone archival.
