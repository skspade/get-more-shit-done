# Phase 17, Plan 01: Implement Health Command

## Goal
Replace the `handleHealth` stub in `cli.cjs` with a real implementation that validates `.planning/` directory structure, config integrity, and state consistency — matching the established pattern from `handleProgress` and `handleTodos`.

## Context
- `handleProgress` and `handleTodos` follow a consistent pattern: `gatherXData()` for data, handler builds rich-mode `message` string, returns `{ command, ...data, message }`
- `cmdValidateHealth()` in `verify.cjs` already performs comprehensive health checks but calls `output()` internally (side effect) — we extract the validation logic into `gatherHealthData()` in `cli.cjs` rather than coupling to `verify.cjs`
- The health command is read-only — no `--repair` flag in the CLI
- Three severity levels: error (broken), warning (degraded), info (informational)
- Overall status: healthy (no errors/warnings), degraded (warnings only), broken (errors present)

<action>
1. Add `gatherHealthData(projectRoot)` function in `cli.cjs` that performs all health checks and returns structured data. Follow the same separation-of-concerns pattern as `gatherProgressData` and `gatherTodosData`.

2. The function checks the following, building `checks`, `errors`, `warnings`, and `info` arrays:

   **Required File Checks (HLTH-01):**
   - `.planning/` directory exists
   - `.planning/PROJECT.md` exists; if yes, verify sections `## What This Is`, `## Core Value`, `## Requirements` are present
   - `.planning/ROADMAP.md` exists
   - `.planning/STATE.md` exists
   - `.planning/config.json` exists
   - `.planning/phases/` directory exists
   - Each check produces an entry in the `checks` array: `{ name, path, passed, detail }`

   **Config Validation (HLTH-02):**
   - If config.json exists, parse it as JSON — report parse errors
   - Validate `model_profile` against allowed values: `['quality', 'balanced', 'budget']`
   - Report unknown top-level keys as info-level notices (not errors)

   **State Consistency (HLTH-03):**
   - Read STATE.md frontmatter using `extractFrontmatter()`
   - Read ROADMAP.md to find which phases are marked complete (`[x]`) and which are not (`[ ]`)
   - Compare STATE.md `current_phase` or phase references against ROADMAP completion status
   - Detect when STATE.md reports a phase as current but ROADMAP shows it completed (or vice versa)
   - Check that STATE.md phase references point to phases that exist on disk

   **Error/Warning Reporting (HLTH-04):**
   - Each issue has: `code` (E001-E005 for errors, W001-W007 for warnings, I001 for info), `message` (human-readable), `fix` (suggested remediation)
   - Use the same code scheme as `cmdValidateHealth` in verify.cjs for consistency

3. Determine overall `status`: `'healthy'` if no errors and no warnings, `'degraded'` if warnings but no errors, `'broken'` if any errors.

4. Return: `{ status, checks, errors, warnings, info }`

5. Replace `handleHealth` stub with real implementation:
   - Call `gatherHealthData(projectRoot)`
   - Build rich-mode formatted string with ANSI colors:
     - Header line: overall status with color (green=healthy, yellow=degraded, red=broken)
     - File checks section: one line per required file with checkmark (pass) or cross (fail)
     - Issues section: errors first (red), then warnings (yellow), each with code + message + fix
     - Summary line: count of errors, warnings, info items
   - ANSI constants: reuse same pattern as handleProgress (BOLD, RESET, GREEN, YELLOW, RED, DIM, CYAN)
   - Return `{ command: 'health', ...data, message }`

**Implementation details to follow:**
- Use `const RED = '\x1b[31m';` for error coloring (not yet defined in other handlers but needed here)
- Pass icon: `\u2713` (checkmark), Fail icon: `\u2717` (ballot x)
- Keep it simple — no premature optimization
- All file reads wrapped in try/catch for graceful handling
</action>
