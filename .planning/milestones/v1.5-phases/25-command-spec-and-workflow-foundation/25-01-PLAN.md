Create `get-shit-done/workflows/brainstorm.md` following the established workflow pattern.

Structure:
```markdown
<purpose>
Run a collaborative brainstorming session: explore project context, ask clarifying questions one at a time, then propose 2-3 distinct approaches with trade-offs and a recommendation.
</purpose>

<process>

## 1. Parse Topic

Extract topic from $ARGUMENTS. If empty, use AskUserQuestion to prompt for a topic:
- header: "Brainstorm"
- question: "What would you like to brainstorm about?"
- (open-ended, no options)

Store as $TOPIC for use throughout.

## 2. Explore Project Context

Read project files to ground the session:
- .planning/PROJECT.md (if exists)
- .planning/REQUIREMENTS.md (if exists)
- .planning/ROADMAP.md (if exists)
- .planning/STATE.md (if exists)
- Recent git commits: `git log --oneline -20`
- Scan codebase using Glob/Grep based on topic keywords for relevant source files

Summarize findings to the user as a brief "Here's what I found" overview. Show:
- Project focus and current state
- Relevant existing code or patterns found
- Any requirements or roadmap items related to the topic

## 3. Ask Clarifying Questions

Ask 3-5 clarifying questions one at a time using AskUserQuestion.

For each question:
- Prefer multiple choice format when options are enumerable
- Use open-ended format when the domain is too broad for pre-set options
- Focus on: purpose/goal, constraints, success criteria, user preferences

Question flow:
1. Purpose/goal clarification — what outcome does the user want?
2. Constraints — what limitations or requirements exist?
3. Success criteria — how will the user know this worked?
4. Preferences — any specific approaches, patterns, or styles preferred?
5. (Optional) Additional context — anything else relevant?

After each answer, adapt subsequent questions based on what was learned.

## 4. Propose Approaches

Based on context exploration and clarifying answers, present 2-3 distinct approaches.

Format each approach as:
```
### Approach {N}: {Name}

{Description — 2-3 sentences explaining the approach}

**Pros:**
- {advantage}
- {advantage}

**Cons:**
- {disadvantage}
- {disadvantage}
```

After listing all approaches, state a recommendation:
```
### Recommendation

I recommend **Approach {N}: {Name}** because {reasoning}.
```

Then use AskUserQuestion:
- header: "Approach Selection"
- question: "Which approach would you like to go with?"
- options: List each approach name plus "Let me suggest modifications"

If user selects an approach, confirm selection.
If user wants modifications, ask what they'd change, then revise and re-present.

## 5. End of Phase 25 Scope

After approach selection, display:
```
Approach selected: {Name}

This brainstorming session is complete. Design presentation and doc writing will be added in a future update.
```

Note: Phase 26 will extend this workflow to present a full design and write it to `.planning/designs/`. Phase 27 will add GSD routing.

</process>
```

Key implementation details:
- The workflow is a single-agent interactive process (not spawning sub-agents)
- AskUserQuestion is the primary interaction mechanism
- Context exploration happens BEFORE any questions (BRAIN-01)
- Questions are one at a time (BRAIN-02)
- Multiple choice preferred when options are enumerable (BRAIN-02)
- 2-3 approaches with pros/cons and recommendation (BRAIN-03)
